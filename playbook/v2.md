# Creator Strategy Playbook — v2

## Global rules

1. **Copy context lines verbatim from the source file.** Do not retype, re-indent, or paraphrase them. A single trailing-space mismatch causes `patch does not apply`.
2. **Count hunk lines precisely.** The `@@` header must show exact old/new line counts. Count removed lines for the first number, added lines for the second, and verify context line counts on both sides.
3. **One hunk, one fix.** Keep the diff to the smallest possible change. If the fix touches two non-adjacent lines, use two hunks with correct intermediate context.
4. **Output strict JSON only.** No markdown fences around the JSON object. No prose before or after. The keys are `root_cause`, `plan`, `reasoning`, `diff`. The `diff` value is a raw unified-diff string with `--- a/` and `+++ b/` headers.
5. **Never invent imports, files, or helper functions.** Fix only the file given.
6. **Verify the fix is syntactically valid** before outputting. If you add `await`, confirm the enclosing function is `async`. If you close a JSX tag, confirm the result parses.

## What failed and why

### diff_apply failures (dominant: ~60% of sampled failures)
**Evidence:** `patch does not apply`, `corrupt patch at line N` across all bug classes — async-002, null-001, offbyone-002, syntax-001, api-001, api-002. The model generates diffs whose context lines, whitespace, or hunk-header line counts do not match the actual source. Even when the *fix logic* is correct, the patch is rejected.

**Root cause:** Context lines are retyped from memory rather than copied verbatim. Hunk `@@` line counts are guessed. Leading spaces vs tabs are swapped.

### build failures (~20%)
**Evidence:** `build failed` for syntax-001, syntax-002, offbyone-001, offbyone-003, null-001. The applied patch introduces a new syntax error or the fix itself is malformed (e.g., adding `>` in the wrong place, breaking JSX structure).

**Root cause:** The fix resolves one syntax issue but creates another, or the edit produces invalid JavaScript/JSX.

### tests failures (~15%)
**Evidence:** Tests fail for async-002, async-004, null-001, offbyone-001, api-001. The patch applies and builds, but the logic is still wrong — e.g., `await` added to the wrong expression, off-by-one fix overshoots, or the API fix doesn't match test expectations.

**Root cause:** The model identifies the bug class but applies the fix to the wrong line or with the wrong operator.

### JSONDecodeError (~5%)
**Evidence:** `creator failed: JSONDecodeError` for api-001. The model wraps JSON in markdown fences or includes trailing prose.

**Root cause:** Output discipline not followed; extra text outside the JSON object.

## Per-bug-class tactics

### syntax
1. Read the file and find the exact unbalanced token — missing `>` on a JSX close tag, unclosed brace, or unterminated string literal.
2. Add only the missing character(s). Do not reformat surrounding lines.
3. After writing the fix, mentally parse the resulting JSX/JS to confirm it is valid.
4. **Copy the two lines above and below the edit point verbatim as context.**

### off-by-one
1. Identify the exact expression with the wrong bound — `+ 1` that should be absent, `<` that should be `<=`, or a slice end that includes one extra element.
2. Change only the operator or the offset. Do not rewrite the entire function.
3. Verify the fix with a concrete example: if `pageSlice(items, 0, 5)` should return 5 items, confirm `end = start + pageSize` yields exactly that.

### null-handling
1. Find the property access that throws on `null`/`undefined`.
2. Add the smallest guard: optional chaining `?.` is preferred when the access chain is the only change needed. Use `?? fallback` when a default is required. Use an early return only if the function must short-circuit.
3. Do not add type checks or logging. One operator change is usually sufficient.

### wrong-api
1. Read the API contract carefully — does `Intl.NumberFormat.format` expect major units or minor units? Does `formatToParts` return `fraction` or `decimal`?
2. Make the single conversion or method swap. For currency, if input is integer cents, divide by 100 before formatting.
3. Check the test expectations to confirm the output format matches.

### async
1. First check the tested function for a missing `await` on a Promise-returning expression.
2. Confirm the enclosing function is already `async`. If not, add `async` to the function declaration — this is a second line change and is acceptable.
3. Place `await` directly before the Promise expression (`Promise.all(...)`, `fetch(...)`, etc.), not before a variable that already holds a resolved value.
4. Do not wrap in `try/catch` unless the test expects error handling.
