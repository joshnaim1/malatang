# Creator Strategy Playbook — v3

## Global rules

1. **Copy context lines character-for-character from the source.** Open the file, locate the exact lines, and transcribe them with identical indentation, trailing spaces, and line endings. A single mismatched space or tab causes `patch does not apply`. This is the #1 failure mode — treat it as the highest priority.
2. **Compute hunk header line counts arithmetically.** Count context + removed lines for the old count; context + added lines for the new count. Do not guess. If the hunk has 3 context, 1 removed, 1 added, the header is `@@ -X,4 +Y,4 @@`.
3. **Use exactly 3 lines of context: one above, the changed line(s), one below.** More context increases the chance of a transcription error. Fewer risks ambiguity. If the line above or below is blank, include it — blank lines are context too.
4. **One hunk, one fix.** Never touch lines unrelated to the bug. If the fix requires two non-adjacent edits, use two hunks with correct intermediate context and accurate cumulative line counts.
5. **Output strict JSON only.** No markdown fences around the JSON. No prose before or after. Keys: `root_cause`, `plan`, `reasoning`, `diff`. The `diff` value is a raw unified-diff string starting with `--- a/` and `+++ b/`.
6. **Never invent imports, files, or helper functions.** Fix only the file given.
7. **Before outputting, re-read the entire function containing your edit and mentally parse it.** Confirm the result is syntactically valid JavaScript/JSX. This catches build failures before they happen.
8. **Read the test expectations before finalizing the fix.** Identify what the test asserts and confirm your fix produces that exact output. This catches logic errors before they happen.

## What failed and why

### diff_apply failures (dominant: ~55% of sampled failures)
**Evidence:** `patch does not apply`, `corrupt patch at line N` across null-001, offbyone-002, syntax-001, api-002, offbyone-003. The fix logic is often correct but the patch is rejected because context lines don't match the source — wrong whitespace, wrong line counts in the `@@` header, or retyped lines with subtle differences.

**Root cause:** Context lines are reconstructed from memory rather than copied verbatim. Hunk line counts are guessed instead of counted. This is the single biggest blocker.

### build failures (~20%)
**Evidence:** `build failed` for syntax-002 (twice), null-004 (twice). The applied patch introduces a new syntax error or the edit produces invalid code — e.g., fixing one JSX tag but breaking another, or changing a value reference incorrectly.

**Root cause:** The model fixes the identified bug without re-parsing the surrounding code. The edit resolves one issue but creates another in the same function.

### tests failures (~20%)
**Evidence:** Tests fail for async-002 (twice), async-004, null-001 (twice), offbyone-002 (three times), api-001, api-002. The patch applies and builds but the logic is still wrong — `await` added to the wrong expression, off-by-one fix overshoots or undershoots, null guard doesn't match test expectations, or API extraction returns the wrong part.

**Root cause:** The model identifies the bug class but doesn't verify the fix against test expectations. It applies a plausible fix without confirming the exact output the test requires.

### JSONDecodeError (~5%)
**Evidence:** `JSONDecodeError` for api-001. The model wraps JSON in markdown fences or includes trailing prose.

**Root cause:** Output discipline broken; extra text outside the JSON object.

## Per-bug-class tactics

### syntax
1. Read the file and find the exact unbalanced token — missing `>` on a JSX close tag, unclosed brace, unterminated string.
2. Add only the missing character(s). Do not reformat or touch any other line.
3. After writing the fix, mentally parse the entire JSX block from the opening tag to the closing tag to confirm balance.
4. Copy the line above and below the edit point verbatim as context — check for tabs vs spaces.

### off-by-one
1. Identify the exact expression with the wrong bound — a `+ 1` that should be absent, a `<` that should be `<=`, or a slice end that includes/excludes one extra element.
2. Change only the operator or the offset constant. Do not rewrite the function.
3. **Plug in the test's input values and trace the arithmetic by hand.** If `pageSlice(items, 0, 5)` should return 5 items, compute `start`, `end`, and `slice(start, end)` manually. Confirm the result count matches the test expectation exactly.
4. If the fix still fails tests, you likely fixed the wrong bound — re-read the test to see which input case fails.

### null-handling
1. Find the property access that throws on `null`/`undefined`.
2. **Read the test to see what it expects when the value is null** — does it expect `undefined`, a fallback string, `0`, or a skipped entry? Match that exactly.
3. Add the smallest guard: optional chaining `?.` when the access chain is the only change. Use `?? fallback` when a default is required. Use an early return only if the test expects short-circuiting.
4. Do not add type checks, logging, or try/catch.

### wrong-api
1. Read the API contract carefully — does `Intl.NumberFormat.format` expect major units or minor units? Does `formatToParts` return `fraction` or `decimal`?
